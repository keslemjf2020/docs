Mega Brain separates everything in your repository into layers based on what it contains and who should have access to it. L1 is the open-source engine that anyone can install via npm. L2 is populated knowledge — dossiers, playbooks, trained agents — distributed only to paying users or through private repositories. L3 is your personal data: inbox files, session logs, and company-specific content that never leaves your machine. Two additional classifications handle secrets (NEVER) and obsolete content (DELETE). Every file in your project belongs to exactly one layer.

Understanding layers matters as soon as you want to distribute anything. The validation pipeline enforces layer boundaries automatically, but you need to classify new files correctly when you create them.

## Quick Reference

| Layer | Name | Git Status | Distribution | Who Uses It |
|---|---|---|---|---|
| **L1** | Community | Tracked | npm (public) | Everyone — open source core |
| **L2** | Premium | Tracked | Private repo (gated) | Paying users |
| **L3** | Personal | Gitignored | Local only / personal backup | Individual user only |
| **NEVER** | Secrets | Gitignored | Never shared anywhere | Nobody |
| **DELETE** | Obsolete | N/A | Remove from repo | N/A |
| **REVIEW** | Unclear | Varies | Needs human decision | Pending classification |

## Layer Definitions

### L1 — Community (npm package)

L1 is the engine that powers Mega Brain for everyone. It ships via `npm install -g mega-brain-ai` and contains no user data, no populated knowledge, and no secrets — only the code and templates needed to run the system.

**What belongs in L1:**
- Core processing engine (`core/`)
- CLI binaries (`bin/`)
- Claude Code integration templates (`.claude/`)
- Conclave agent templates (`agents/conclave/`)
- Agent template files (`agents/_templates/`)
- Documentation (`docs/`)
- Empty structure markers (`.gitkeep` files anywhere in the repo)

```
core/                          → L1 (Core engine)
core/tasks/HO-TP-001.md        → L1 (Core engine)
bin/validate-package.js        → L1 (Core engine)
.claude/hooks/skill_router.py  → L1 (Core engine)
agents/conclave/               → L1 (Core engine)
docs/LAYERS.md                 → L1 (Core engine)
inbox/.gitkeep                 → L1 (Empty structure marker)
```

<Tip>
  `.gitkeep` files are **always L1**, regardless of which directory they live in. They contain no data — they just hold a directory's place in git.
</Tip>

**What L1 does NOT include:**
- Populated knowledge bases or dossiers (L2)
- User data or session history (L3)
- API keys or credentials (NEVER)

Community users can generate L2-equivalent content themselves by running the full pipeline on their own materials. L2 distribution simply provides pre-built content.

### L2 — Premium (populated content)

L2 contains the actual knowledge generated by the Mega Brain pipeline: trained agent personalities, curated dossiers, domain playbooks, and extracted DNA. This content is the result of running materials through the system and represents significant processing investment.

**What belongs in L2:**
- Populated mind clone agents (`agents/minds/` with content)
- Populated cargo agents (`agents/cargo/` with profiles)
- Knowledge dossiers (`knowledge/dossiers/`)
- Knowledge playbooks (`knowledge/playbooks/`)
- DNA knowledge base (`knowledge/dna/`)
- Knowledge sources (`knowledge/sources/`)
- Pipeline artifacts with extracted insights

```
agents/minds/ALEX-HORMOZI/             → L2 (Premium content)
agents/cargo/AGENT-CFO/                → L2 (Premium content)
knowledge/dossiers/persons/ALEX.md     → L2 (Premium content)
knowledge/playbooks/SALES.md           → L2 (Premium content)
knowledge/dna/                         → L2 (Premium content)
artifacts/insights/                    → L2 (Premium content)
```

<Note>
  L2 is a **superset of L1**. When you distribute L2 content, you include all L1 content plus the populated knowledge on top of it.
</Note>

**Accessing L2:**

```bash
# Clone the premium private repo (requires authentication)
git clone https://private-repo/mega-brain-premium.git

# Or sync from cloud storage
mega-brain sync --from premium
```

### L3 — Personal (never distributed)

L3 is everything your pipeline produces that is specific to you: the raw materials you ingest, the processing logs, session history, and your company's data. This content is gitignored by default and is never committed to any shared repository.

**What belongs in L3:**
- Raw input materials (`inbox/`)
- Processing and session logs (`logs/`)
- Claude Code session history (`.claude/sessions/`)
- Mission control state (`.claude/mission-control/`)
- Company-specific data (`agents/sua-empresa/`)

```
inbox/                         → L3 (Personal data)
inbox/alex-hormozi/video.txt   → L3 (Personal data)
logs/                          → L3 (Personal data)
logs/batches/BATCH-001.md      → L3 (Personal data)
.claude/sessions/              → L3 (Personal data)
.claude/mission-control/       → L3 (Personal data)
agents/sua-empresa/            → L3 (Personal data)
```

L3 content is valuable to you but meaningless to others. You are responsible for your own backups:

```bash
# Optional personal backup
rsync -av .claude/sessions/ ~/Dropbox/mega-brain-backup/sessions/
rsync -av inbox/ ~/Dropbox/mega-brain-backup/inbox/
rsync -av logs/ ~/Dropbox/mega-brain-backup/logs/
```

### NEVER — Secrets and Credentials

NEVER files must never be committed to any git repository — not even a private one. If a file contains an API key, token, password, or certificate, it is NEVER.

**What belongs in NEVER:**

```
.env                           → NEVER (Secrets)
.env.local                     → NEVER (Secrets)
.mcp.json                      → NEVER (Secrets/sensitive config)
credentials.json               → NEVER (Secrets)
token.json                     → NEVER (Secrets)
settings.local.json            → NEVER (Secrets)
*.key                          → NEVER (Certificate)
*.pem                          → NEVER (Certificate)
```

<Warning>
  When in doubt, classify a file as NEVER. A false NEVER classification costs you nothing. A missed secret in a public package can never be fully revoked.
</Warning>

### DELETE — Obsolete Content

DELETE marks files that were once useful but have been superseded by a newer implementation. Schedule them for removal with `git rm`.

```
archive/*/finance-agent/       → DELETE (Replaced by agents/cargo/)
archive/*/talent-agent/        → DELETE (Replaced by agents/cargo/)
```

Run `python3 core/intelligence/audit_layers.py` to get the current list of delete candidates.

### REVIEW — Needs Human Classification

REVIEW is not a permanent state — it's a signal to stop and decide. The automated classifier assigns REVIEW to anything it can't confidently place. In the Phase 7 audit, 12,183 items (58.6%) were REVIEW, primarily IDE configs, planning documents, and root-level project files.

```
.cursor/                       → REVIEW (IDE config)
.planning/                     → REVIEW (Planning docs)
README.md                      → REVIEW (Root readme)
package.json                   → REVIEW (Project config)
```

## Classification Rules

### Decision Flowchart

Use this flowchart when you create a new file and need to assign it a layer:

```
New file — which layer?
│
├── Contains secrets, API keys, tokens, or credentials?
│   YES → NEVER
│   NO  ↓
│
├── Superseded, abandoned, or explicitly marked obsolete?
│   YES → DELETE
│   NO  ↓
│
├── User-generated content (inbox, logs, sessions, company data)?
│   YES → L3
│   NO  ↓
│
├── Populated knowledge (dossiers, playbooks, DNA, artifacts)?
│   YES → L2
│   NO  ↓
│
├── A .gitkeep file?
│   YES → L1 (Empty structure marker — always L1)
│   NO  ↓
│
├── Core engine, CLI, Claude integration, or documentation?
│   YES → L1
│   NO  ↓
│
└── Cannot determine → REVIEW
```

### Classification Criteria at a Glance

| Criteria | L1 | L2 | L3 | NEVER | DELETE | REVIEW |
|---|---|---|---|---|---|---|
| Contains API keys or tokens | | | | ✅ | | |
| Obsolete/replaced implementation | | | | | ✅ | |
| User's pipeline output (personal) | | | ✅ | | | |
| User-specific business content | | | ✅ | | | |
| Populated knowledge (dossiers, playbooks) | | ✅ | | | | |
| Core engine code | ✅ | | | | | |
| Empty structure marker (`.gitkeep`) | ✅ | | | | | |
| Cannot determine without context | | | | | | ✅ |

## How to Classify a New File

<Steps>
  <Step title="Check NEVER patterns first">
    Does the filename match `.env`, `*.key`, `*.pem`, `credentials.json`, or `settings.local.json`? Does the content include API keys, tokens, or passwords?

    **If yes → NEVER.** Add to `.gitignore` immediately.
  </Step>
  <Step title="Check DELETE patterns">
    Is this file or directory in the `DELETE_PATTERNS` list in `audit_layers.py`? Is it explicitly superseded by a newer implementation?

    **If yes → DELETE.** Schedule a `git rm`.
  </Step>
  <Step title="Check L3 patterns">
    Does the path start with `inbox/`, `logs/`, `.claude/sessions/`, or `agents/sua-empresa/`?

    **If yes → L3** (unless it's a `.gitkeep`, which is always L1).
  </Step>
  <Step title="Check L2 patterns">
    Does the path start with `agents/minds/`, `agents/cargo/` (with content), `knowledge/dossiers/`, or `artifacts/insights/`?

    **If yes → L2** (unless it's a `.gitkeep` or empty directory).
  </Step>
  <Step title="Check L1 patterns">
    Does the path start with `core/`, `bin/`, `.claude/`, `agents/conclave/`, or `docs/`? Is it a `.gitkeep` file anywhere in the repo?

    **If yes → L1.**
  </Step>
  <Step title="Assign REVIEW if none match">
    If none of the above matched, classify as **REVIEW**. Open a discussion, consult the criteria table, make a decision, and add the pattern to `audit_layers.py` to prevent future ambiguity.
  </Step>
</Steps>

## Programmatic Classification

Run `audit_layers.py` any time you want a full picture of layer compliance across the entire repository:

```bash
# Generate complete audit report
python3 core/intelligence/audit_layers.py

# Output:
# - docs/audit/AUDIT-REPORT.json  (machine-readable, 20,797+ items)
# - docs/audit/AUDIT-REPORT.md    (human-readable summary)
```

<CodeGroup>
```bash View delete candidates
python3 core/intelligence/audit_layers.py
jq '.delete_candidates' docs/audit/AUDIT-REPORT.json
```

```bash Classify a single file
python3 -c "
from pathlib import Path
import sys
sys.path.insert(0, 'core/intelligence')
from audit_layers import classify_path
layer, reason = classify_path(Path('agents/cargo/AGENT-CFO/AGENT.md'), Path('.'))
print(f'{layer}: {reason}')
"
```
</CodeGroup>

To add or update a classification rule, edit the relevant `*_PATTERNS` list in `core/intelligence/audit_layers.py` and re-run the audit.

## Validation

Mega Brain enforces layer compliance through three tools. See [Validation](/advanced/validation) for the full documentation.

<Steps>
  <Step title="Package validation">
    Confirms only L1 files are in the npm package:
    ```bash
    node bin/validate-package.js
    # Expected: PASSED: All 247 pack files are L1
    ```
  </Step>
  <Step title="Pre-publish gate">
    Blocks publish if non-L1 content or secrets are detected:
    ```bash
    node bin/pre-publish-gate.js
    # Runs automatically via prepublishOnly
    ```
  </Step>
  <Step title="Full audit">
    Generates a complete classification report of every file in the repo:
    ```bash
    python3 core/intelligence/audit_layers.py
    cat docs/audit/AUDIT-REPORT.md
    ```
  </Step>
</Steps>

## Troubleshooting

<Accordion title="File classified into the wrong layer">
1. Check whether the path matches any pattern in `audit_layers.py`.
2. Verify `.gitkeep` files are not being incorrectly overridden.
3. Update the pattern lists if your project adds new directories.
4. Re-run the audit to confirm the fix.
</Accordion>

<Accordion title="Package contains non-L1 files">
1. Run `validate-package.js --json` to see which files are flagged and why.
2. Move L2/L3 files to their correct location.
3. Add the offending paths to `.gitignore`.
4. Update the `files` field in `package.json` if needed.
5. Re-run validation to confirm.
</Accordion>

<Accordion title="Secrets detected in the package">
1. Identify the exact file from the gate output.
2. Remove the secret from the file, or move the file to NEVER classification.
3. Ensure the file is in `.gitignore`.
4. If the file was already committed, run `git rm --cached <file>` and force-push.
5. Re-run `pre-publish-gate.js` to confirm the issue is resolved.
</Accordion>

<Accordion title="Too many REVIEW items">
1. Run the audit to get a breakdown of REVIEW by directory.
2. Classify the most common patterns first (IDE configs, planning dirs).
3. Add those patterns to `audit_layers.py` so they're handled automatically next time.
4. Document each classification decision in the pattern file.
5. Re-run the audit — REVIEW count should drop significantly.
</Accordion>
