Not everything in your Mega Brain workspace should be shared with everyone. Your core engine is open source and belongs to the community. Your processed knowledge and mind clone agents are premium content you've earned by running the pipeline. Your raw source materials and session logs are private and personal. The layer system gives you a simple, enforceable classification for every file — so you always know what can be published, what stays gated, and what never leaves your machine.

## Quick Reference

| Layer | Name | Git Status | Distribution | Audience |
|---|---|---|---|---|
| **L1** | Community | Tracked | npm package (public) | Everyone — open source core |
| **L2** | Premium | Tracked (private repo) | Private repository (gated) | Paying users |
| **L3** | Personal | Gitignored | Local only / personal backup | You only |

<Info>
  L2 is a superset of L1. When you distribute an L2 package, it includes all L1 content plus the populated knowledge content. L3 content never appears in either distribution.
</Info>

## Layer 1: Community

**The open-source engine that powers Mega Brain.** Distributed via npm as `mega-brain-ai`, L1 contains everything someone needs to build their own knowledge management system — without any of your personal content.

  <Card title="Claude Code Integration" icon="plug">

  <Card title="Templates & Structure" icon="file">
    `agents/_templates/`, `agents/conclave/`, `docs/`, and all `.gitkeep` structure markers

**What belongs in L1:**
```
core/                             → L1 (Core engine)
core/tasks/                       → L1 (Core engine)
core/intelligence/audit_layers.py → L1 (Core engine)
bin/                              → L1 (CLI tooling)
.claude/                          → L1 (Claude integration)
.claude/rules/RULE-GROUP-1.md     → L1 (Operational rules)
agents/conclave/                  → L1 (Deliberation templates)
docs/                             → L1 (Documentation)
inbox/.gitkeep                    → L1 (Empty structure marker)
```

<Tip>
  `.gitkeep` files are always L1, regardless of which directory they live in. They mark where populated content will go without containing any personal data.
</Tip>

## Special Categories

<Warning>
  **NEVER** — Some files must never be committed to any repository, regardless of layer.

  This includes `.env`, `.env.local`, `credentials.json`, `token.json`, `.mcp.json`, and any `*.key` or `*.pem` certificate files. These contain API keys and tokens that would expose your accounts if published. Add them to `.gitignore` permanently.
</Warning>

There is also a **DELETE** category for files that have been superseded by newer implementations. Run `python3 core/intelligence/audit_layers.py` to identify delete candidates in your workspace.

## Classification Decision Flow

When you're not sure which layer a file belongs in, work through these questions in order:

<Steps>
  <Step title="Check NEVER first">
    Does the file contain API keys, tokens, passwords, or credentials? If yes → **NEVER** (add to `.gitignore` immediately).
  </Step>
  <Step title="Check for L3 signals">
    Is it user-generated content — raw materials in `inbox/`, processing logs, session history, or company-specific data? If yes → **L3** (gitignored).
  </Step>
  <Step title="Check for L2 signals">
    Is it populated knowledge content — dossiers, playbooks, DNA schemas, pipeline artifacts, or populated agent files? If yes → **L2** (private repository).
  </Step>
  <Step title="Check for .gitkeep">
    Is it a `.gitkeep` file anywhere in the project? If yes → **L1** (structure marker, always public).
  </Step>
  <Step title="Default to L1">
    Is it part of the core engine, CLI tooling, Claude integration, or template system? If yes → **L1** (public repository).
  </Step>
</Steps>

## Validating Layer Classification

Before publishing or pushing to any repository, validate your layer classification:

```bash
# Verify only L1 content is staged for the public package
npm run validate:layers

# Full pre-publish gate: secrets scan + layer validation
node bin/pre-publish-gate.js

# Programmatic audit with detailed output
python3 core/intelligence/audit_layers.py
```

## Best Practices

<Steps>
  <Step title="Separate concerns clearly">
    Keep your L1 engine, L2 knowledge content, and L3 personal data in clearly separated directories with separate git configurations.
  </Step>
  <Step title="Never mix layers">
    Do not commit L3 content into L1 or L2 repositories. Do not include secrets anywhere in any repository.
  </Step>
  <Step title="Validate before every publish">
    Always run the pre-publish gate before pushing to any public or premium repository.
  </Step>
  <Step title="Document new additions">
    When you add new content or directories, record which layer they belong to and why — especially for edge cases.
  </Step>
</Steps>
